GitHub + Hexo 博客写作全攻略：常用命令、排版技巧与部署经验

很多人搭建个人博客时，都会选择 Hexo + GitHub Pages 这一套组合。原因很简单：免费、轻量、部署方便，而且非常适合程序员和喜欢折腾的人。

我自己在使用 Hexo 写博客的过程中，踩过不少坑，也慢慢总结出了一套比较顺手的写作流程。这篇文章就把 Hexo 博客写作时常用的命令、排版技巧、资源管理方法，以及部署到 GitHub 的经验 全部整理一下，方便以后查阅，也希望能帮到刚开始接触 Hexo 的朋友。

一、Hexo 是什么，为什么适合搭配 GitHub

Hexo 是一个基于 Node.js 的静态博客框架。简单理解，它会把你写好的 Markdown 文件生成静态网页，然后你可以把这些网页部署到 GitHub Pages 上。

这一套方案的优点很明显：

  • 写作时只需要专注 Markdown 内容
  • 页面生成速度快
  • 适合技术博客、学习笔记、生活记录
  • 可以和 Git 配合做版本管理
  • 部署到 GitHub Pages 的成本几乎为零

如果你本身就熟悉 GitHub，那么 Hexo 会非常顺手。

三、一篇 Hexo 文章通常长什么样

Hexo 的文章本质上就是 Markdown 文件，不过文件头部通常会带有一段 Front-matter 配置。

这些字段里最常用的是：

  • title：文章标题
  • date：发布时间
  • tags：标签
  • categories：分类
  • description：文章摘要或 SEO 描述

如果主题支持，还可以加入：

  • cover：封面图
  • top：是否置顶
  • mathjax：是否开启数学公式
  • comments：是否开启评论

6. 标签不要太多，分类不要太碎

这是很多人写博客时容易忽略的问题。

建议：

  • 分类控制在大方向，例如 前端、后端、工具、随笔
  • 标签用于补充关键词，例如 Hexo、GitHub Pages、Markdown
  • 一篇文章 2 到 5 个标签通常就够用了

如果标签和分类太细，后期管理会非常混乱。

1. 页面改了但是没生效

2. 部署后 GitHub Pages 没更新

可以重点检查：

  • deploy.repo 是否写对
  • 分支名是不是 main 或 master
  • 是否安装了 hexo-deployer-git

3. 图片显示不出来

常见原因包括：

  • 图片路径写错
  • 图片没有放到正确目录
  • 使用了本地绝对路径

八、适合长期坚持的几个小建议

除了会用命令，更重要的是形成稳定的输出习惯。

我自己的建议是：

  • 每篇文章只讲清楚一个核心主题
  • 标题尽量具体，不要太空泛
  • 技术文章多配命令和示例
  • 发布前至少本地预览一次
  • 尽量让文件名、标签、分类保持一致风格

博客写久了你会发现，Hexo 不只是一个静态博客工具，它更像是一个帮助你沉淀知识的仓库。
