FastAPI 学习笔记：从第一个接口到可部署服务

1. FastAPI 解决什么问题

FastAPI 是一个面向 HTTP API 的 Python Web 框架，核心特点是：

  • 用 Python 类型标注描述输入和输出；
  • 自动完成请求解析、数据校验、序列化和 OpenAPI 文档；
  • 基于 ASGI，可同时处理普通 def 与异步 async def 路由；
  • 依赖注入让数据库会话、认证、配置等横切逻辑可复用。

它不是“让任何程序都自动更快”的框架。async 在网络、数据库等 I/O 等待场景能提升并发利用率；CPU 密集型任务仍应交给任务队列、进程池或独立计算服务。

3. 路径、查询与请求体参数

FastAPI 通过函数签名判断参数来源：路径模板中出现的同名参数是路径参数；简单类型通常是查询参数；Pydantic 模型通常是 JSON 请求体。

常见规则：

  • GET /todos?skip=0&limit=20 中的 skip、limit 是查询参数；
  • GET /todos/{todo_id} 中的 todo_id 是路径参数，转换失败会得到 422；
  • JSON 传给 TodoCreate，字段缺失或不合法也会得到结构化的 422 错误；
  • 路由声明顺序有意义：/users/me 应写在 /users/{user_id} 之前。

4. Pydantic：校验与序列化的边界

请求模型（DTO）不应直接等同数据库实体。一个实用的分层是：

  • TodoCreate：创建时允许的字段；
  • TodoUpdate：更新时允许且可选的字段；
  • TodoRead：响应中允许暴露的字段；
  • ORM 实体：持久化结构，不直接作为外部契约。

在路由中声明 response_model=TodoRead。这不仅生成文档，也会过滤意外返回的敏感字段，例如 password_hash、内部标识或审计信息。

model_dump() 用于模型转字典，model_validate() 用于按模型规则构造/校验对象。Pydantic v2 的旧版 dict()、parse_obj() 写法仍可能看到，但新代码优先使用 v2 API。

5. 响应、状态码与错误处理

成功响应应使用语义正确的状态码：创建为 201，无响应体的删除为 204，非法输入由校验层返回 422，未认证 401，无权限 403，不存在 404，冲突 409。

不要把底层数据库异常原样返回给客户端；记录完整日志，对外给稳定、可理解且不泄露实现细节的错误信息。

6. 依赖注入：复用请求级逻辑

Depends 将路由依赖显式写进签名。它适合分页参数、数据库会话、当前用户、权限检查和配置等逻辑。

依赖可以嵌套。认证依赖获得当前用户，业务路由再依赖它；框架会在单个请求内缓存默认依赖结果。需要每次重新计算时使用 Depends(..., use_cache=False)。

8. 项目组织与路由拆分

完成单文件 CRUD 并接入持久化后，再把代码按职责拆分；这样目录中的每一层都有明确来处。不要把所有路由、数据库和认证逻辑堆到 main.py。

版本前缀使破坏性 API 变更可以并行发布，例如保留 /api/v1，新增 /api/v2。不要把数据库 Session 或 Web 框架对象扩散到核心业务层，服务层更易测试和替换。

9. 认证、授权与安全基线

认证回答“你是谁”，授权回答“你能做什么”。FastAPI 可通过 fastapi.security 集成 OAuth2 Bearer Token，以下是路由层的最小形态：

安全要点：

  • 密码使用 Argon2 或 bcrypt 等专用哈希，不保存明文，也不自行实现哈希算法；
  • JWT 只签发必要声明，校验签名、过期时间、签发方和受众；密钥放在环境变量或密钥服务；
  • 对每个资源做对象级授权，例如待办必须属于当前用户，不能只校验“已登录”；
  • CORS 仅列出可信前端源，携带 Cookie 时不可使用通配源；
  • 限制上传类型/大小，统一日志脱敏，并为登录、写操作设置限流和审计；
  • HTTPS 在反向代理或应用入口终止，生产环境关闭 --reload 和调试信息。

10. 异步的正确使用方式

工作类型 推荐写法 原因
普通 CPU 计算或同步 ORM def 路由 FastAPI 会在线程池执行同步路径函数
async HTTP、WebSocket、AsyncSession async def + await 等待 I/O 时可让出事件循环
大文件解析、图片处理、模型推理 队列/进程池/独立服务 避免阻塞 Web 进程和事件循环

async def 内使用 requests、同步数据库驱动或 time.sleep() 会阻塞事件循环。异步 HTTP 客户端可使用 httpx.AsyncClient；延时使用 await asyncio.sleep()。并发不是无限资源，仍需要连接池、超时、限流和下游熔断策略。

11. 中间件、生命周期与后台任务

中间件适合请求 ID、耗时、统一安全响应头等 HTTP 横切能力。BackgroundTasks 只适合短小、可丢失的进程内工作，例如发通知；邮件批量发送、文件转码、重试任务应使用 Celery、RQ、Arq 或消息队列。

应用初始化和释放资源使用 lifespan。例如启动时建立共享 HTTP 客户端，停止时关闭；不要每次请求都创建昂贵的连接池。

12. 文件、表单、WebSocket 与流式响应

文件和表单使用 File、UploadFile、Form。UploadFile 比一次性读入 bytes 更适合大文件；上传前后都要校验扩展名、MIME、大小和实际内容。

WebSocket 是长连接协议，认证、心跳、断连清理、消息大小限制和订阅授权不能省略。SSE 或 StreamingResponse 更适合服务端单向推送，例如 LLM 文本流。

13. 测试：先测 HTTP 合约，再测业务规则

FastAPI 的 TestClient 可在不启动真实服务器时测试 HTTP 行为。测试中覆盖依赖，可以替换真实数据库、外部支付或认证服务。

建议至少覆盖：成功路径、422 参数错误、401/403、404、数据库事务回滚、分页边界和关键业务幂等性。对 get_db 覆盖为测试事务或临时数据库，测试结束后回滚，避免测试污染开发数据。

14. 配置、日志、观测与部署

配置应随环境改变，不应硬编码在代码里。可用 pydantic-settings 从环境变量和 .env 加载，并将密钥排除在 Git 之外。

部署检查清单：

  • 使用 uvicorn app.main:app --host 0.0.0.0 --port 8000 或由 Gunicorn/Uvicorn worker 管理进程；
  • 放在 Nginx、Traefik 或云负载均衡器之后，正确处理 X-Forwarded-* 代理头；
  • 提供 /healthz、/readyz，区分进程存活和依赖就绪；
  • 记录结构化日志，关联 request ID，监控错误率、延迟、吞吐量和数据库连接池；
  • 对数据库迁移、密钥轮换、备份恢复和依赖漏洞建立发布流程；
  • 容器化时以非 root 用户运行，固定依赖版本，区分构建期与运行期配置。
